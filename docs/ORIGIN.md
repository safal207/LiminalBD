# Origin of LiminalDB

LiminalDB began from a simple question:

> What kind of database is needed for systems operating under uncertainty, pressure, and continuous transition?

Traditional databases store state.
LiminalDB was designed for systems where state is alive: changing, reacting, adapting, and leaving traces that can be replayed and understood.

Its core metaphors — cells, impulses, reflexes, salience, seed gardens, mirror timelines, and variant manifolds — come from the idea that memory is not passive storage. In adaptive systems, memory is an active participant in decision-making.

LiminalDB therefore treats storage as more than persistence. It is a reactive memory substrate for systems that must remain coherent under pressure.

## Design intuition

LiminalDB is built around five founding intuitions:

1. **State is not enough.** A system also needs signals, transitions, and reasons.
2. **Memory should be replayable.** Important decisions must leave an auditable trace.
3. **Load changes meaning.** Under stress, a system should adapt rather than simply fail.
4. **Not all events are equal.** Salience matters; some signals deserve stronger memory.
5. **Future paths matter.** Adaptive systems often need to hold multiple possible variants before committing.

## Why it exists

LiminalDB exists for agentic and adaptive software systems that need more than CRUD:

- append-only memory,
- replayable timelines,
- reactive cells,
- adaptive feedback loops,
- goal-oriented task lifecycles,
- and explainable transitions.

It is a database/runtime for systems that do not merely store state, but survive transition.
